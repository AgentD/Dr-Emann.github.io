Squashfs Binary Format (WIP)

A squashfs filesystem consists of a maximum of nine parts, packed together on a byte alignment:

Superblock
Important information about the archive, including locations of other sections
Compression Options
If non-default compression options have been used, then these are stored here.
Datablocks & Fragments
The contents of the files in the archive, split into blocks
Inode Table
Metadata (ownership, permissions, etc) for items in the archive
Directory Table
Directory listings, including file names, and references to inodes
Fragment Table
Description of fragment locations within the Datablocks & Fragments section
Export Table
A mapping from inode numbers to disk locations, required for NFS export
UID/GID Lookup Table
A list of UID/GIDs. IDs are referenced as an index into this table
Xattr Table
Xattrs for items in the archive

The superblock

The superblock is the first section of a squashfs archive, and contains important information about the archive, including the locations of other sections of the archive.

Offset Name Type Description
0x00 magic u32 Must match the value of 0x73717368 to be considered a squashfs archive
0x04 inode_count u32 The number of inodes stored in the inode table
0x08 creation_epoch_time i32 The number of seconds (not counting leap seconds) since 00:00, Jan 1 1970 UTC when the archive was created (or last appended to)
0x0C block_size u32 The size of a data block in bytes. Must be a power of two between 4096 and 1048576 (1 MiB)
0x10 fragment_entry_count u32 The number of entries in the fragment table
0x14 compression_id u16
1 - GZIP
2 - LZMA
3 - LZO
4 - XZ
5 - LZ4
6 - ZSTD
0x16 block_log u16 The log2 of block_size. If block_size and block_log do not agree, the archive is considered corrupt
0x18 flags u16
(Flags)		 See Superblock Flags
0x1A id_count u16 The number of entries in the id lookup table
0x1C version_major u16 The major version of the squashfs file format. Should always equal 4
0x1E version_minor u16 The minor version of the squashfs file format. Should always equal 0
0x20 root_inode_ref u64
(InodeRef)		 A reference to the inode of the root directory of the archive
0x28 bytes_used u64 The number of bytes used by the archive. Because squashfs archives are often padded to 4KiB, this can often be less than the file size
0x30 id_table_start u64 The byte offset at which the id table starts
0x38 xattr_id_table_start u64 The byte offset at which the xattr id table starts
0x40 inode_table_start u64 The byte offset at which the inode table starts
0x48 directory_table_start u64 The byte offset at which the directory table starts
0x50 fragment_table_start u64 The byte offset at which the fragment table starts
0x58 export_table_start u64 The byte offset at which the export table starts

Superblock Flags

Value Name Description
0x0001 UNCOMPRESSED_INODES Inodes are stored uncompressed. For backward compatibility reasons, UID/GIDs are also stored uncompressed.
0x0002 UNCOMPRESSED_DATA Data are stored uncompressed
0x0004 CHECK Unused in squashfs 4+. Should always be unset
0x0008 UNCOMPRESSED_FRAGMENTS Fragments are stored uncompressed
0x0010 NO_FRAGMENTS Fragments are not used. Files smaller than the block size are stored in a full block.
0x0020 ALWAYS_FRAGMENTS If the last block of a file is smaller than the block size, it will be instead stored as a fragment
0x0040 DUPLICATES Identical files are recognized, and stored only once
0x0080 EXPORTABLE Filesystem has support for export via NFS (The export table is populated)
0x0100 UNCOMPRESSED_XATTRS Xattrs are stored uncompressed
0x0200 NO_XATTRS Xattrs are not stored
0x0400 COMPRESSOR_OPTIONS The compression options section is present
0x0800 UNCOMPRESSED_IDS UID/GIDs are stored uncompressed. Note that the UNCOMPRESSED_INODES flag also has this effect. If that flag is set, this flag has no effect. This flag is currently only available on master in git, no released version of squashfs yet supports it.

Metadata Blocks

Metadata blocks are compressed in 8KiB blocks. A metadata block is prefixed by a u16 header. The highest bit of the header is set if the block is stored uncompressed (this will happen if the block grew when compressed, or e.g. the UNCOMPRESSED_INODES superblock flag is set). The lower 15 bits specifies the size of the metadata block (not including the header) on disk.

To read a metadata block, read a u16. If the highest bit is set (size & 0x8000 == 0x8000) the following data is uncompressed. Mask out the highest bit to get the size of the block data on disk (this should always be <= 8KiB). Read that many bytes. If the data is compressed, uncompress the data. In pseudocode: 

header = read_u16(offset=offset)
data_size = header & 0x7FFF
compressed = header & 0x8000
data = read(offset=offset+2, len=data_size)
if(compressed) {
    data = uncompress(data)
}
return data

Neither the size on disk, nor the compressed size should exceed 8KiB. The uncompressed size should always be equal to 8KiB, with the exception of the last metadata block of a section, which may have an uncompressed size less than 8KiB.

Compression Options

If the COMPRESSOR_OPTIONS flag is set, this section will be present immidately after the superblock, otherwise this section will not be present. If this section is present, it consists of a single metadata block, which is always uncompressed. The data is interpreted differently based on the compressor (compression_id).

LZMA

LZMA does not support any compression options

GZIP

Offset Name Type Description
0x00 compression_level i32 Should be in range 1…9 (inclusive). Defaults to 9.
0x04 window_size i16 Should be in range 8…15 (inclusive) Defaults to 15.
0x08 strategies i16 A bitfield describing the enabled strategies. If no flags are set, the default strategy is implicitly used. Flags:
0x01 Default
0x02 Filtered
0x04 Huffman Only
0x08 Run Length Encoded
0x10 Fixed

XZ

Offset Name Type Description
0x00 dictionary_size i32 Should be > 8KiB, and must be either the sum of a power of two, or the sum of two sequential powers of two (2n or 2n + 2n+1)
0x04 executable_filters i32 A bitfield describing the additional enabled filters attempted to better compress executable code. Flags:
0x01 x86
0x02 powerpc
0x04 ia64
0x08 arm
0x10 armthumb
0x20 sparc

LZ4

Offset Name Type Description
0x00 version i32 The only supported value is 1 (LZ4_LEGACY)
0x04 flags i32 A bitfield describing the enabled LZ4 flags. There is currently only one possible flag:
0x01 Use LZ4 High Compression(HC) mode

ZSTD

Offset Name Type Description
0x00 compression_level i32 Should be in range 1..22 (inclusive). The real maximum is the zstd defined ZSTD_maxCLevel()

Fragment Table

Fragments are combined into fragment blocks of at most block_size bytes long. This table describes the location and size of these fragment blocks, not the fragments within them.

This table is stored in two levels: The fragment block entries are stored in metadata blocks, and the file offsets to these metadata blocks are stored at the offset specified by the fragment_table_start field of the superblock.

Each metadata block can store 512 fragment block entries (16 bytes per fragment block entry), so there will be ceil(fragment_entry_count / 512.0) metadata blocks (and the same number of u64 offsets stored at fragment_table_start)

To read the list of fragment block entries, read ceil(fragment_entry_count / 512.0) u64 offsets starting at fragment_table_start, then read the metadata blocks at the offsets read, interpreting the data of the metadata blocks as a packed array of fragment block entries.

Fragment Block Entry

Offset Name Type Description
0x00 start u64 The offset within the archive where the fragment block starts
0x08 size u32 This stores two pieces of information. If the block is uncompressed, the 0x1000000 (1<<24) bit wil be set. The remaining bits describe the size of the fragment block on disk. Because the max value of block_size is 1 MiB (1<<20), and the size of a fragment block should be less than block_size, the uncompressed bit will never be set by the size.
0x0C _unused u32 This field is unused

UID/GID Table

UID/GIDs are both stored as u32s. Both UIDs and GIDs are treated as IDs, if a file is owned by user 1000, and group 1000, the ID 1000 will be stored only once. UID/GIDs in inodes are stored as u16 indexes into this table (and thus, a maximum of 65535 unique IDs may be stored).

This table is stored in two levels: The IDs are stored in metadata blocks, and the file offsets to these metadata blocks are stored at the offset specified by the id_table_start field of the superblock.

Each metadata block can store 2048 UID/GIDs (4 bytes per ID), so there will be ceil(id_count / 2048.0) metadata blocks used for IDs (and the same number of u64 offsets stored at id_table_start).

To read the list of IDs, read ceil(id_count / 2048.0) u64 offsets starting at id_table_start, then read the metadata blocks at the offsets read.
